# Ασφάλεια δεδομένων – ανεβασμένα αρχεία (ATLAS Lite)

## Τι ισχύει στην πράξη

### Πού πηγαίνουν τα δεδομένα;

Όταν ο χρήστης ανεβάζει PDF μέσω του **Streamlit** (`st.file_uploader`):

1. **Το αρχείο στέλνεται από το browser στον server** όπου τρέχει η Streamlit εφαρμογή (π.χ. Streamlit Community Cloud ή δικό σας server). Δεν μένει μόνο στον browser.
2. **Στον server** το αρχείο κρατιέται σε **μνήμη (RAM)** σε αντικείμενο τύπου BytesIO, όχι σε μόνιμο δίσκο.
3. Κατά την επεξεργασία, η συνάρτηση `extract_efka_data` γράφει **προσωρινά** ολόκληρο το PDF σε ένα προσωρινό αρχείο στο δίσκο του server, το διαβάζει για εξαγωγή πινάκων και στο τέλος **το διαγράφει** (`os.unlink`).

Άρα: τα δεδομένα **περνούν** από τον server (RAM και προσωρινά και στο δίσκο μέχρι να τελειώσει η επεξεργασία). Δεν αποθηκεύονται μόνιμα σε αρχείο στον server.

### Πότε «φεύγουν» από τον server;

Σύμφωνα με την Streamlit:

- Όταν ο χρήστης **κλείσει** την καρτέλα του browser όπου άνοιξε την εφαρμογή.
- Όταν **αφαιρέσει** το αρχείο από το file uploader (clear).
- Όταν **ανεβάσει άλλο** αρχείο (το προηγούμενο αντικαθίσταται).

Μέχρι τότε, το ανεβασμένο αρχείο (raw bytes) παραμένει στη **session** του χρήστη στη μνήμη του server. Σε μεγάλα αρχεία έχουν αναφερθεί περιπτώσεις όπου η απελευθέρωση μνήμης γίνεται πλήρως όταν κλείσει το Streamlit instance (π.χ. restart εφαρμογής).

---

## Η πρώτη σελίδα του PDF (προσωπικά δεδομένα)

- **Η εφαρμογή δεν διαβάζει/εξάγει περιεχόμενο από την πρώτη σελίδα.**  
  Στο `extract_tables_adaptive` (app_final.py) υπάρχει ρητό:
  - `if page_num < 1: continue`  
  δηλαδή η **πρώτη σελίδα (page_num 0) παραλείπεται** και δεν εξάγονται πίνακες ή κείμενο από αυτή.

- **Ωστόσο**, ο **ολόκληρος όγκος του PDF** (συμπεριλαμβανομένης της πρώτης σελίδας) είναι:
  - στη μνήμη του server κατά το upload και κατά τη διάρκεια της session,
  - προσωρινά και στο δίσκο κατά την `extract_efka_data` (μέχρι να τρέξει το `finally` και να γίνει `os.unlink`).

Δηλαδή: η εφαρμογή **δεν χρησιμοποιεί** τα προσωπικά δεδομένα της πρώτης σελίδας για ανάλυση ή εμφάνιση, αλλά τα **bytes** της πρώτης σελίδας περνούν από τον server και υπάρχουν προσωρινά στη μνήμη και στο temp αρχείο.

## Διόρθωση μηνύματος στην εφαρμογή

Στην εφαρμογή (app_lite.py) εμφανίζεται:

> «Τα δεδομένα επεξεργάζονται αποκλειστικά στον browser σας (client-side) και δεν αποθηκεύονται σε κανέναν server.»

Αυτό **δεν** αντιστοιχεί στην πραγματικότητα: το αρχείο **στέλνεται στον server** και επεξεργάζεται εκεί (RAM + προσωρινό αρχείο που διαγράφεται). Συνιστάται να αλλάξει το κείμενο ώστε να αναφέρει ότι:

- το αρχείο επεξεργάζεται στον server,
- δεν αποθηκεύεται μόνιμα,
- διαγράφεται από τη μνήμη όταν κλείσει η session ή αφαιρεθεί το αρχείο.

Ένα παράδειγμα (μπορείτε να το προσαρμόσετε):

- «Τα αρχεία που ανεβάζετε στέλνονται στον server για επεξεργασία. Δεν αποθηκεύονται μόνιμα· διαγράφονται όταν κλείσετε τη σελίδα ή αφαιρέσετε/αντικαταστήσετε το αρχείο.»

## Μπορεί ένας χρήστης να δει το αρχείο ενός άλλου;

**Κατά σχεδιασμό: όχι.** Το Streamlit χρησιμοποιεί **session state** ανά σύνδεση: κάθε browser session (κάθε χρήστης, ή κάθε καρτέλα που ανοίγει ξεχωριστά την εφαρμογή) έχει δικό του `st.session_state`. Το ανεβασμένο αρχείο και τα εξαγόμενα δεδομένα κρατούνται στο `st.session_state["lite_uploaded_file"]` και `st.session_state["lite_df"]`, δηλαδή **μόνο** στη session αυτού του χρήστη. Ο χρήστης Β δεν έχει πρόσβαση στο session state του χρήστη Α, οπότε δεν μπορεί να δει το αρχείο ή τα αποτελέσματα του Α.

**Σημαντική επιφύλαξη:** Έχουν αναφερθεί σε forums και issues περιπτώσεις όπου σε **ορισμένα deployed περιβάλλοντα** (π.χ. Community Cloud ή συγκεκριμένες ρυθμίσεις) το `st.session_state` εμφάνιζε **λανθασμένα κοινή** κατάσταση μεταξύ διαφορετικών χρηστών/καρτελών. Δηλαδή θεωρείται bug και όχι σχεδιασμός, αλλά το ρίσκο υπάρχει σε production. Για ευαίσθητα προσωπικά δεδομένα, αξίζει να το έχετε υπόψη και να ενημερώνεστε για διορθώσεις της Streamlit.

**Σύνοψη:** Στην πράξη συνήθως κάθε χρήστης βλέπει μόνο το δικό του αρχείο. Το ρίσκο «να δει ο ένας το αρχείο του άλλου» βασίζεται σε λανθασμένη κοινή χρήση session state, που δεν θα έπρεπε να συμβαίνει αλλά έχει αναφερθεί σε ορισμένα deploy.

## Σύνοψη

| Ερώτηση | Απάντηση |
|--------|----------|
| Διαγράφονται τα ανεβασμένα δεδομένα; | Ναι, από τη μνήμη του server όταν κλείνει η session / αφαιρείται το αρχείο. Το προσωρινό αρχείο στο δίσκο διαγράφεται αμέσως μετά την επεξεργασία. |
| Παραμένουν κάπου μόνιμα; | Όχι· καμία μόνιμη αποθήκευση σε αρχείο στον server για τα uploads. |
| Η πρώτη σελίδα διαβάζεται από την εφαρμογή; | Όχι· η κώδικας την παραλείπει (`page_num < 1: continue`) και δεν εξάγει πίνακες ή κείμενο από αυτή. |
| Περνά όμως η πρώτη σελίδα από τον server; | Ναι· ολόκληρο το PDF (συμπ. πρώτη σελίδα) είναι προσωρινά στη μνήμη και στο temp αρχείο κατά την επεξεργασία. |
| Μπορεί ο χρήστης Β να δει το αρχείο του χρήστη Α; | Κατά σχεδιασμό όχι (session state ανά χρήστη). Έχουν όμως αναφερθεί bugs σε ορισμένα deploy όπου η session state μοιράζονταν λανθασμένα. |
